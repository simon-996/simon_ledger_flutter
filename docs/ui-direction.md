# Simon Ledger UI 方向文档

> 后续新增页面、弹窗、组件和交互时先阅读本文档。目标是让 Simon Ledger 保持统一、精致、流畅，而不是每个功能单独设计一套视觉。

## 产品气质

Simon Ledger 是高频使用的多人账本工具。设计目标是：

- 清晰：用户能快速知道当前账本、金额、参与人、同步状态。
- 安静：界面服务于记账和对账，不做强装饰、重渐变、大面积营销式视觉。
- 可信：离线优先和同步状态要明确，让用户知道数据已经保存在本机。
- 精致：卡片边界、文字层级、触摸反馈和动效要稳定一致。
- 轻快：高频操作路径少停顿，动画丝滑但不抢操作节奏。

关键词：**Apple Calm、低干扰、清晰层级、轻量动效、离线可信、多人协作可见**。

## 颜色规则

颜色必须优先来自 `AppTheme` 和 `Theme.of(context).colorScheme`。

整体视觉走 Apple 式冷白灰底色，不使用重渐变、霓虹高光和大面积彩色背景。页面应该像一套安静的系统工具：背景轻、边界细、主操作清楚。

固定语义：

- 主色 `AppTheme.primaryColor`：确认、同步成功、主要操作、支出记账强调色。
- 收入强调色：使用交易表单里的柔和暖红 `transactionIncomeAccentColor`，不要使用高饱和红色。
- 危险/错误 `AppTheme.errorColor`：删除、失败、错误。不要再把普通支出当成错误红展示。
- 成功/共享 `AppTheme.successColor`：共享成员、成功状态。
- 警告 `AppTheme.tertiaryColor`：待同步、弱网、需注意状态。
- 信息 `AppTheme.infoColor`：普通提示、说明性状态。
- 图表颜色使用 `AppTheme.chartColors`，这是少数允许多色的场景，但必须保持低饱和、同一灰度气质，不要在统计页直接使用彩虹色或随机色。

页面背景层级：

- 页面底色：`colorScheme.surface`。
- 普通内容卡片：`surfaceContainerLowest`，白色或接近白色。
- 浮层、底部导航和顶部提醒：可使用半透明白色 + 极细边界，体现轻玻璃质感。
- 可点击列表项：优先用留白、细边界和轻微按压反馈，不要每个项目都做厚重卡片。
- 选中项：`primaryContainer` 背景 + `primary` 细边框，透明度要低。

不要新增裸 `Colors.green/orange/purple/cyan` 等业务色。确实需要新增颜色时，先放进 `AppTheme` 并命名为语义用途。

## 形状与间距

推荐圆角：

- 12：紧凑提示、状态条、内部小容器。
- 16：输入框、选择项、分类、币种、人员卡片。
- 24：主要内容卡片，符合 Apple 式柔和大圆角。
- 28-32：弹窗、大型容器。
- 999：胶囊标签、人员 chip。

推荐间距：

- 页面左右边距：`AppTheme.pagePadding`，当前为 16。
- 卡片内部：14-16。
- 同组控件：8-10。
- 不同模块：14-16。
- 页面底部留白：至少 24，避免贴近底部导航或安全区。

## 文字规则

文案要短、直接、面向用户。

统一术语：

- 本机账本：只保存在当前设备。
- 待同步：已保存在本机，等待上传云端。
- 已同步：本机和云端一致。
- 同步失败：数据仍在本机，但暂时无法上传。
- 共享账本：多人加入的云端账本。
- 共同钱包：所有使用人员共同承担。
- 个人垫付：某人先付，其他人应付给他。

按钮文字优先使用动词：

- 保存、同步、重试、复制、加入、退出、删除。
- 避免“确定”承载复杂含义，删除/退出等危险操作必须写清动作。

错误信息不展示技术细节，统一走 `FriendlyError` 或 `AppNotice` 的用户可读文案。

## 动效规则

动效必须优先使用 `AppMotion`：

- `AppMotion.fast`：按钮选中态、chip 状态、轻量反馈。
- `AppMotion.normal`：列表项、卡片状态、页面局部切换。
- `AppMotion.slow`：首屏进入、成功反馈、较强的视觉反馈。
- `AppMotion.standard`：普通状态变化。
- `AppMotion.emphasized`：进入、弹出、成功反馈。

高频操作不要使用过强弹跳。成功记账、生成邀请、同步完成可以有更强反馈，但持续时间要短。

Apple Calm 动效原则：

- 只动画 `opacity` 和 `transform`，不要动画高度、宽度、top、left。
- 点击态必须有轻微物理反馈，例如 `scale 0.98` 或阴影收缩。
- 页面切换保持轻微位移和淡入，不做大幅滑动。
- 同步、保存、分享使用整块区域的轻量状态过渡，不堆多个小 spinner。

列表动效：

- 首屏列表可使用 `AppAnimatedEntry` 做轻微延迟进入。
- 搜索过滤、选中态、同步状态使用轻量 `AnimatedContainer` 或 `AppAnimatedSwitcher`。
- 不要在同一区域叠加多层 `AnimatedSize + AnimatedSwitcher`，避免高度抖动和卡顿。

弹窗：

- 底部弹窗高度要稳定，动态内容在内部滚动或轻量淡入。
- 删除/退出等强确认使用居中对话框。
- 长流程如邀请加入、导入本地账本使用页面或完整底部弹窗，不挤在小弹窗里。

## 组件规则

优先使用现有公共组件：

- `AppSectionCard`：内容卡片，默认使用 Apple Calm 白色浮层、24 圆角、细边界和轻阴影。
- `AppSectionHeader`：模块标题。
- `AppEmptyState`：空状态。
- `AppLoadingState` / `AppInlineLoadingCard`：加载态。
- `AppNotice`：顶部提醒。
- `AppAnimatedEntry` / `AppAnimatedSwitcher`：进入和切换。
- `AppLedgerPeopleChips`：账本成员展示。
- `AppPersonChoiceGrid`：人员选择。
- `TransactionTypeSelector`、`CurrencySelector`、`CategorySelector`：记账表单选择。

新增组件时要先判断是否能扩展现有组件，避免重复造局部样式。

## 页面方向

### 记账页

这是最高频页面。原则：

- 首屏优先展示当前账本、收入/支出、金额、币种。
- 收入/支出、金额、币种应合并为一个 Apple 式主输入浮层，使用白色、28 圆角、细边界和轻阴影，避免首屏出现多个碎片卡片。
- 账本选择要快：支持搜索、顺序与账本列表一致。
- 金额输入和保存按钮必须稳定，不被动态区域挤压。
- “共同钱包/个人垫付”和人员选择的出现消失要流畅，不导致页面跳动。

### 账本列表

原则：

- 最新或用户排序靠前的账本在上方。
- 账本名称是主信息，编号弱化。
- 成员头像昵称可见，共享成员用成功色边框区分。
- 同步和分享操作要有明确加载态，避免重复点击。
- 账本卡片使用 Apple 式白色浮层：28 圆角、极细边界、轻阴影；同步/分享时整卡变为轻状态面，不叠加多余按钮动画。

### 账本详情

原则：

- 明细记录易扫读，金额和币种清楚。
- 人员筛选不应造成头像跳动。
- 删除或编辑流水后本地立即反馈。

### 统计页

原则：

- 统计页重点是看懂结果，不是展示所有明细。
- 顶部筛选要紧凑稳定。
- 图表使用 `AppTheme.chartColors`。
- 代付结算要清楚表达“谁应付给谁”和金额。

### 我的页面

原则：

- 分为账户资料、同步中心、共享/导入三个信息区域。
- 账户资料卡像 iOS 设置页账户入口：整卡可点击、头像明显、昵称最大、账号弱化，右侧箭头只做低存在感提示。
- 同步状态必须明确区分：无待同步、待同步、同步中、同步失败。
- 操作成功/失败使用顶部 `AppNotice`，不要堆叠长时间弹窗。

## 新功能接入检查清单

- 是否使用 `AppTheme`/`ColorScheme`，没有新增裸色值？
- 是否使用 `AppMotion`，没有随意新增动画时长和曲线？
- 是否复用了公共组件？
- 文案是否使用统一术语？
- 三端窄屏和宽屏是否不会溢出？
- 离线状态是否有本地完成反馈？
- 同步、保存、删除等异步操作是否有加载态和防重复点击？
- 空状态、错误状态、无网状态是否有用户能理解的说明？
